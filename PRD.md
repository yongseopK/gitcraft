# GitCraft PRD
### GitHub 코딩 스타일 분석 서비스

> 버전 0.1 | 작성일 2026-04-11

---

## 1. 프로젝트 개요

### 한 줄 정의
GitHub 커밋 데이터를 분석해서 "당신의 개발자 유형"을 알려주는 서비스

### 핵심 가치 제안
TypeCraft가 "당신이 생각하는 나"라면, GitCraft는 **"코드가 증명하는 나"**다.
자기 보고 방식의 설문이 아닌 실제 행동 데이터 기반이라 결과에 설득력이 있고,
"나 이런 사람이었어?" 하는 발견의 재미가 있다.

### 왜 지금인가
- 개발자 커뮤니티에서 자기 정체성에 대한 관심이 높아지는 추세 (GitHub Wrapped 바이럴 사례)
- GitHub API가 완전 공개라 별도 데이터 수집 인프라 불필요
- TypeCraft로 검증된 "개발자 대상 바이럴 포맷" 노하우 보유

---

## 2. 문제 정의

### 사용자 페인포인트
개발자들은 자신의 코딩 습관을 어렴풋이 알지만, 실제로 데이터로 보여주는 서비스가 없다.

- "나 야근을 얼마나 하나?" → 커밋 시간대를 보면 알 수 있는데 직접 분석하는 사람은 없음
- "나 꾸준히 코딩하나 몰아치나?" → 느낌은 있지만 수치로 본 적 없음
- "나 다른 개발자랑 비교하면 어떤 편이지?" → 비교 기준이 없음

### 기존 대안의 한계

| 서비스 | 한계 |
|---|---|
| GitHub Wrapped (Netlify) | 연 1회, 단순 통계 나열, 한국어 없음 |
| CoderStats | 유형화 없이 숫자만, 재미 없음 |
| WakaTime | IDE 플러그인 설치 필요, 진입장벽 높음 |
| TypeCraft | 설문 기반, 실제 데이터 반영 안 됨 |

**공백**: 실제 GitHub 데이터 기반으로 개발자 "유형"을 뽑아주는 서비스가 없다.

## 3. 타겟 사용자

### 1차 타겟 (코어)
- **주니어~미드레벨 개발자** (경력 1~5년)
- GitHub 공개 레포 10개 이상 보유
- SNS 공유에 거부감 없는 연령대 (20대 중반~30대 초반)
- 개발자 커뮤니티에 속해있음 (오픈카톡, Discord, LinkedIn)

### 2차 타겟 (확장)
- 취준생 / 부트캠프 수료생: 자신의 GitHub 활동을 객관적으로 보고 싶음
- 시니어 개발자: 후배나 팀원 분석해서 재미로 공유
- 채용 담당자: 지원자 GitHub 빠르게 파악 (B2B 확장 가능)

## 4. 핵심 기능 (MVP)

### F1. GitHub 유저네임 입력 & 분석
- 텍스트 입력 한 번으로 분석 시작
- OAuth 없이 공개 데이터만으로 분석 (진입장벽 최소화)
- 분석 대상: 최근 6개월 / 상위 10개 레포 / 공개 커밋

### F2. 유형 판정 & 결과 화면
- 12가지 개발자 유형 중 1개 판정
- 유형명 + 부제목 + 설명 텍스트
- 핵심 지표 4개 시각화 (활동 시간대, 언어, 리듬, 완성도)
- 커밋 히트맵 (실제 GitHub 잔디 스타일)

### F3. 공유
- 카카오톡 공유 (동적 OG 이미지 - 유형명 + 대표 통계)
- URL 복사
- 공유된 링크로 들어온 사람도 결과 + "나도 분석하기" CTA

### F4. 재분석 / 비교
- 다른 GitHub 유저네임으로 재분석 (친구 분석)
- 두 유저 나란히 비교 ("이 팀의 케미는?") — MVP 이후

## 6. 분석 알고리즘

### 수집 데이터

```
GitHub Public API (인증 불필요)
├── GET /users/{username}                     → 기본 프로필
├── GET /users/{username}/repos?per_page=100  → 레포 목록 (fork 여부 포함)
├── GET /repos/{owner}/{repo}/commits         → 커밋 목록 + 시간 + 메시지
└── GET /repos/{owner}/{repo}/languages       → 언어 사용 비율

※ fork 비율은 repos 응답의 fork 필드로 계산 (추가 req 불필요)
※ 분석 대상: fork=false 레포 중 pushed_at 최근 순 상위 10개
```

### 5개 분석 축 + 2개 특수 조건

#### 축 1. 활동 시간대 (When)
커밋 timestamp를 로컬 시간 기준으로 4개 구간 비율 계산
(GitHub API commit.author.date는 UTC → 한국 서비스이므로 UTC+9 변환)

| 결과 | 판정 조건 |
|---|---|
| 새벽형 | 00:00~05:59 비율 ≥ 25% |
| 오전형 | 06:00~11:59 비율이 4구간 중 최다 AND ≥ 30% |
| 오후형 | 12:00~17:59 비율이 4구간 중 최다 AND ≥ 30% |
| 야간형 | 18:00~23:59 비율 ≥ 45% |
| 불규칙형 | 위 조건 모두 미해당 |

> 우선순위: 새벽형 > 야간형 > 오전형 > 오후형 > 불규칙형
> (새벽형과 야간형은 비율 기준이라 다른 구간 최다와 겹칠 수 있으므로 먼저 체크)

#### 축 2. 커밋 리듬 (Rhythm)
분석 기간(최근 26주)을 주 단위로 나눠 주간 커밋 수 배열 생성

| 결과 | 판정 조건 |
|---|---|
| 꾸준형 | 활동 주간(커밋 1개+)이 전체의 60%+ AND 주간 커밋 수 표준편차 ≤ 평균의 50% |
| 몰아치기형 | 상위 20% 주간의 커밋 합이 전체의 60%+ |
| 마감형 | 레포별로 커밋 시계열을 3등분했을 때 마지막 1/3 구간 커밋이 전체의 55%+ 인 레포 비율 ≥ 50% |
| 불규칙형 | 위 조건 모두 미해당 |

> 우선순위: 꾸준형 > 마감형 > 몰아치기형 > 불규칙형

#### 축 3. 전문성 방향 (Scope)
전체 레포의 언어 바이트 수를 합산해 비율 계산

| 결과 | 판정 조건 |
|---|---|
| 풀스택형 | 프론트 언어군 합계 ≥ 20% AND 백엔드 언어군 합계 ≥ 20% |
| 전문가형 | Top 2 언어 합계 ≥ 80% (풀스택형 미해당 시) |
| 탐험가형 | 사용 언어 종류 ≥ 5개 AND 전문가형 미해당 |

```
프론트 언어군: JavaScript, TypeScript, HTML, CSS, Vue, Svelte
백엔드 언어군: Python, Java, Go, Rust, C#, PHP, Ruby, Kotlin, C, C++
```

> 우선순위: 풀스택형 > 전문가형 > 탐험가형
> (풀스택형은 전문가형 조건을 덮어씀)

#### 축 4. 소통 방식 (Communication)
분석 대상 커밋 메시지 전체 기준

| 결과 | 판정 조건 |
|---|---|
| 컨벤션파형 | `^(feat\|fix\|docs\|style\|refactor\|test\|chore)(\(.+\))?:` 패턴 ≥ 50% |
| 이모지형 | 메시지에 이모지 포함 커밋 ≥ 30% (컨벤션파형 미해당 시) |
| 설명충형 | 메시지 평균 길이 ≥ 40자 (한글 1자 = 1자 기준) |
| 요약파형 | 메시지 평균 길이 ≤ 15자 |
| 보통형 | 위 조건 모두 미해당 |

> 우선순위: 컨벤션파형 > 이모지형 > 설명충형 > 요약파형 > 보통형

#### 축 5. 완성도 (Completion)

| 결과 | 판정 조건 |
|---|---|
| 완성형 | 레포당 평균 커밋 ≥ 30개 AND 커밋 5개 이하 레포 비율 ≤ 30% |
| 시작형 | 커밋 5개 이하 레포 비율 ≥ 50% |
| 개선형 | 생성일 기준 1년 이상 된 레포 중 최근 3개월 내 커밋이 있는 레포 비율 ≥ 40% |
| 보통형 | 위 조건 모두 미해당 |

> 우선순위: 완성형 > 개선형 > 시작형 > 보통형

#### 특수 조건 A. 주말 집중도 (Weekend)
- **주말형**: (토요일 + 일요일) 커밋 합계 ≥ 55%
- 평일/주말 구분은 커밋 timestamp UTC+9 기준 요일로 계산

#### 특수 조건 B. 오픈소스 기여도 (OpenSource)
- **오픈소스형**: 전체 레포 중 fork 레포 비율 ≥ 35%
- repos API 응답의 `fork: true` 필드로 판별 (추가 req 없음)

## 7. 유형 시스템 (12가지)

5개 축 + 2개 특수 조건 → 우선순위 기반 매핑으로 12개 유형 중 1개 판정
(유형명은 재미와 공유 욕구를 위해 과장된 캐릭터로 작성)

### 유형 목록

| # | 유형명 | 부제 | 판정 조건 |
|---|---|---|---|
| 1 | 🌙 새벽 감성 코더 | 세상이 잠든 시간에 코딩하는 낭만주의자 | 새벽형 + (꾸준형 OR 개선형) |
| 2 | 🔥 마감 직전 스프린터 | 평소엔 잠잠, D-1에 폭발 | (마감형 OR 몰아치기형) + (불규칙형 OR 야간형) |
| 3 | ⚙️ 9 to 6 클린 코더 | 퇴근 후엔 진짜 쉬는 건강한 개발자 | (오전형 OR 오후형) + 꾸준형 |
| 4 | 🌊 주말 해커 | 평일은 생존, 주말은 창조 | 특수조건A: 주말형 |
| 5 | 🗺️ 언어 수집가 | 새 언어 보면 일단 Hello World부터 | 탐험가형 + 시작형 + 언어 종류 ≥ 6 |
| 6 | 🔩 한 우물 장인 | 하나를 파도 제대로 판다 | 전문가형 + 완성형 |
| 7 | 📦 아이디어 부자 | 레포는 많은데 main이 없다 | 시작형 + (탐험가형 OR 보통형) |
| 8 | 🏗️ 풀스택 건축가 | 프론트도 백엔드도 내 영역 | 풀스택형 + 완성형 |
| 9 | 📝 문서화 신봉자 | README 없으면 코드가 아니다 | 설명충형 + 완성형 |
| 10 | ⚡ 침묵의 스나이퍼 | 커밋은 적지만 하나하나 크다 | 요약파형 + 완성형 |
| 11 | 🌐 오픈소스 시민 | 혼자보단 함께, Fork와 PR의 달인 | 특수조건B: 오픈소스형 |
| 12 | 🎯 컨벤션 원리주의자 | feat: fix: docs: 없으면 커밋 못 한다 | 컨벤션파형 + 꾸준형 |

### 분류기 판정 로직 (우선순위 순)

```
function classify(axes, special):

  // 특수 조건 먼저 체크 (행동 패턴이 가장 명확)
  if special.isOpenSource:   return TYPE_11  // 🌐 오픈소스 시민
  if special.isWeekend:      return TYPE_4   // 🌊 주말 해커

  // 시간대 × 리듬 조합
  if when==새벽 AND rhythm in [꾸준,개선]:    return TYPE_1   // 🌙 새벽 감성 코더
  if when in [오전,오후] AND rhythm==꾸준:    return TYPE_3   // ⚙️ 9 to 6 클린 코더
  if rhythm in [마감,몰아치기] AND when in [야간,불규칙]: return TYPE_2  // 🔥 마감 직전 스프린터

  // 전문성 × 완성도 조합
  if scope==풀스택 AND completion==완성:      return TYPE_8   // 🏗️ 풀스택 건축가
  if scope==전문가 AND completion==완성:      return TYPE_6   // 🔩 한 우물 장인

  // 소통 방식 × 완성도 조합
  if comm==설명충 AND completion==완성:       return TYPE_9   // 📝 문서화 신봉자
  if comm==요약파 AND completion==완성:       return TYPE_10  // ⚡ 침묵의 스나이퍼
  if comm==컨벤션 AND rhythm==꾸준:           return TYPE_12  // 🎯 컨벤션 원리주의자

  // 탐험가 × 완성도 조합 (5 vs 7 분기)
  if scope==탐험가 AND completion==시작 AND languageCount >= 6:
                                              return TYPE_5   // 🗺️ 언어 수집가
  if completion==시작:                        return TYPE_7   // 📦 아이디어 부자

  // 폴백: 남은 새벽/야간 조합
  if when==새벽:                              return TYPE_1   // 🌙 새벽 감성 코더
  if when in [오전,오후]:                     return TYPE_3   // ⚙️ 9 to 6 클린 코더

  // 최종 폴백
  return TYPE_2  // 🔥 마감 직전 스프린터 (불규칙 패턴의 기본값)
```

### 유형 5 vs. 7 분기 기준

두 유형 모두 "시작형 + 탐험가형" 조합이나 핵심 동기가 다름.

| | 🗺️ 언어 수집가 | 📦 아이디어 부자 |
|---|---|---|
| 핵심 동기 | 언어/기술 탐색 | 아이디어 실현 시도 |
| 판정 핵심 | 언어 종류 ≥ 6 | 언어 종류 < 6, 레포 수 많음 |
| 레포 특성 | 레포마다 주력 언어가 다름 | 같은 언어로 프로젝트를 여럿 시작 |

## 8. 엣지 케이스 처리

| 상황 | 처리 방안 |
|---|---|
| 공개 레포 0개 | "분석 가능한 공개 활동이 없어요" 안내 + 공개 레포 만들기 링크 |
| 총 커밋 20개 미만 | 제한된 데이터 분석 가능하나 정확도 낮음 경고 표시 |
| 레포가 fork만인 경우 | fork 커밋 제외 후 재분석, 오리지널 커밋 없으면 별도 안내 |
| Organization 계정 | 개인 계정 분석만 지원, 안내 메시지 |
| GitHub API Rate Limit 초과 | 요청 큐잉 + 대기 안내 or GitHub OAuth 유도 |
| 비공개 활동만 있는 사람 | "잔디는 있는데 분석할 공개 데이터가 없어요" |

## 9. 기술 스펙

### 스택

```
Frontend + BFF: Next.js (Pages Router) — Vercel 배포
  ├── /pages/index.js          — 유저네임 입력
  ├── /pages/result/[username].js  — 결과 (공유 URL)
  └── /pages/api/analyze.js    — GitHub API 호출 + 분석 로직

캐싱: Upstash Redis (Vercel KV)
  └── key: username:YYYYMMDD → 24시간 TTL

OG 이미지: @vercel/og (동적 생성)
  └── /pages/api/og.js → 유형명 + 대표 통계

선택: Claude API
  └── 유형 설명 텍스트 개인화 (고정 텍스트 대신 실제 데이터 반영)
```

### GitHub API 호출 전략

```
1단계: /users/{username}                             → 1 req
         → 기본 프로필, public_repos 수 확인

2단계: /users/{username}/repos?per_page=100&type=owner → 1 req
         → 전체 레포 목록 (fork 필드 포함)
         → fork 비율 계산 (특수조건B)
         → fork=false 레포 중 pushed_at 최근 순 상위 10개 선택

3단계: /repos/{owner}/{repo}/commits?per_page=100    → 최대 10 req
         → 커밋 시간 + 메시지 수집
         → 병렬 요청 (Promise.all)

4단계: /repos/{owner}/{repo}/languages               → 최대 10 req
         → 언어 바이트 수 수집
         → 병렬 요청 (Promise.all)

총 22 req / 분석 1회 (3·4단계는 병렬이라 실제 대기 시간 단축)
→ 비인증 60 req/h → 시간당 약 2.7명 분석 가능
→ Server-side PAT 설정 시 5,000 req/h → 227명/h
→ GitHub OAuth 도입 시 유저별 5,000 req/h
```

**결론**: MVP는 서버사이드 PAT(환경변수)로 시작. 비인증은 트래픽 생기는 즉시 서비스 불가 수준이므로 처음부터 PAT 필수.

### 캐싱 전략
- Redis에 `{username}:{YYYYMMDD}` 키로 결과 저장
- 24시간 TTL (하루 한 번 이상 재분석 불필요)
- 캐시 히트 시 GitHub API 호출 0번

### 분석 처리 위치
- Next.js API Route에서 서버사이드로 처리
- 클라이언트에 GitHub 토큰 노출 없음
- Vercel Edge Function 사용 가능 (응답속도 개선)
